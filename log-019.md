# Log 019: Organizing Geometry.lean

## Introduction
Designing a formal geometry library involves more than selecting mathematical definitions and theorems. The internal organization of the implementation also influences readability, maintainability, and future development. Although Lean accepts many different file structures, a long-term research project benefits from a consistent internal architecture. This log describes the organizational principles adopted for the central implementation file `Geometry.lean`.

## A Mathematical Rather Than Historical Organization
The structure of the library should reflect mathematical dependencies rather than the chronological history of the project. The order in which research logs are written is not necessarily the order in which declarations should appear inside the Lean library. New mathematical insights may require earlier definitions to be refined or additional lemmas to be inserted. The implementation therefore follows the logical structure of synthetic geometry instead of the historical sequence of the research diary.

## Hierarchical Organization
The current organization of `Geometry.lean` follows four principal layers:
1. **Definitions** — primitive mathematical concepts.
2. **Axioms** — accepted foundations of the geometry.
3. **Derived Lemmas** — reusable intermediate results.
4. **Main Theorems** — complete formal developments.

Within each layer, declarations are grouped by mathematical subject rather than by the research log in which they first appeared.

## Continuous Evolution
The library is intended to evolve continuously. New definitions may be introduced, existing proofs simplified, auxiliary lemmas extracted, and declarations reordered whenever this improves the mathematical organization. Such changes represent architectural refinement rather than changes to the mathematics itself.

## Stable Documentation
Because the implementation is expected to evolve, the project deliberately separates documentation from code:
* Research logs explain mathematical ideas.
* `Geometry.md` records identifiers, dependencies, and implementation metadata.
* `Geometry.lean` contains the formal implementation.

This separation allows the source code to be reorganized freely while preserving stable documentation and references.

## Research Perspective
The objective is not to preserve the history of implementation but to construct a coherent mathematical library. The implementation should resemble a well-organized textbook rather than a chronological development diary. As the project grows, this architectural discipline becomes increasingly important for both maintainability and future formal developments.
