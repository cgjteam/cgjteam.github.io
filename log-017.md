# Log 017 — Stable References in a Formal Mathematics Project

### Introduction

As a formal mathematics project evolves, its implementation inevitably changes. Definitions are refined, proofs are optimized, declarations are renamed, and sections of code are reorganized. These modifications improve the quality of the library, but they also create a documentation problem: how can research articles continue to refer reliably to mathematical objects when the source code is constantly changing?

This question led to one of the fundamental architectural decisions of the CGJteam Lab project.

### The Problem

A direct reference to a location inside a source file is fragile.

For example, a research log may point to a declaration located on line 350 of a Lean file. After inserting several new lemmas near the beginning of the file, the same declaration may move to line 520 without any mathematical change. Every reference based on line numbers immediately becomes obsolete.

The same problem appears when a large proof is reorganized or partially refactored.

The implementation evolves; the mathematics does not.

### Stable Mathematical Identifiers

To solve this problem, every formal mathematical object receives a permanent identifier.

Examples include:

* DEF-001
* AXM-001
* LEM-004
* THM-001

These identifiers are independent of

* source code locations,
* declaration order,
* file organization,
* proof optimization.

The identifier refers to the mathematical object rather than to its current implementation.

### The Role of Geometry.md

The Geometry Library Index serves as the central registry of these identifiers.

For every mathematical object it records

* its permanent identifier,
* the Lean declaration,
* mathematical dependencies,
* implementation status,
* related research logs.

Research logs therefore reference mathematical identifiers rather than implementation details.

Geometry.md then connects these identifiers to the current Lean implementation.

### Advantages

This architecture provides several important benefits.

* Research articles remain valid after code refactoring.
* Geometry.lean may be reorganized without breaking documentation.
* The documentation reflects mathematical structure rather than implementation history.
* Future tools can automatically generate dependency graphs directly from the identifier system.

### Research Perspective

One of the objectives of formal mathematics is long-term maintainability.

A proof assistant verifies the current implementation, but a research project must also preserve stable mathematical references that remain meaningful as the implementation evolves.

Separating permanent mathematical identifiers from implementation details is therefore an essential part of the project's architecture.
